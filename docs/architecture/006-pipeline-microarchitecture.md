# ADR 006: Конвеєрна мікроархітектура

Дата: 2026-07-18
Статус: Прийнято

## Контекст

ADR 001–005 визначили ISA (набір інструкцій, формати, opcode-простір) і функціональну семантику кожної інструкції. Не визначено, як інструкції виконуються в часі. Поточний симулятор (`Cpu::step()`) виконує повний цикл fetch-decode-execute-writeback за один виклик — це зручно для перевірки коректності семантики, але не відображає жодної реальної мікроархітектури.

## Рішення

### Модель виконання: 5-стадійний конвеєр
| Стадія | Дія |
|---|---|
| IF | `fetchWord(pc)`, обчислення `pc+4` |
| ID | `decode(raw)`, читання `rs1`/`rs2` з регістрового файлу |
| EX | ALU-операція, обчислення адреси (`rs1+imm`), обчислення умови переходу |
| MEM | `load`/`store` до пам'яті (для решти інструкцій — прохідна стадія) |
| WB | запис результату в регістровий файл |

У кожен такт до 5 інструкцій одночасно перебувають на різних стадіях. Регістровий файл потребує 2 read-портів (rs1+rs2 одночасно в ID) і 1 write-порту (WB) — конфліктів ресурсів (structural hazard) немає.

### Data hazard (RAW): forwarding

Результат з виходу EX або MEM подається напряму на вхід EX наступної інструкції (EX→EX та MEM→EX forwarding paths), минаючи запис-і-читання регістрового файлу. Це усуває stall для найпоширенішого випадку — залежної арифметики підряд (`ADD x3, x1, x2` одразу після інструкцій, що пишуть x1/x2).

**Важливий нюанс, який forwarding не покриває:** load-use hazard. Дані від `load` фізично з'являються лише в кінці стадії MEM — якщо наступна інструкція одразу використовує це значення в EX, forward немає звідки взяти (дані ще не прийшли з пам'яті). ### Control hazard: predict-not-taken + flush

Конвеєр завжди продовжує вибірку лінійно (`pc+4`), не чекаючи результату умови переходу. Умова обчислюється в EX. Якщо перехід фактично мав відбутись:
- дві вже вибрані інструкції (в IF і ID) скидаються (flush — перетворюються на bubble/nop)
- `pc` перезаписується на цільову адресу
- штраф: 2 такти на кожен взятий перехід

Обрано замість завжди-stall (простіше, але дорожче для програм з рідкісними переходами вперед) і замість branch predictor (складніше, передчасна оптимізація для першої робочої версії конвеєра).

## Наслідки

- `Cpu` перестає бути "виконати інструкцію за виклик" — стає "виконати один такт" (`tick()`), з явними pipeline-регістрами між стадіями (IF/ID, ID/EX, EX/MEM, MEM/WB), що зберігають проміжний стан
- Існуючий `step()` (одна інструкція) і `run()` можна зберегти як зручні обгортки над `tick()` для GUI/тестів, що не потребують циклового рівня деталізації
- Знадобиться hazard detection unit (для load-use stall) і forwarding unit (для EX/MEM → EX) як окремі логічні блоки в `Cpu`
- GUI (Step/Run) можна розширити режимом "1 такт" на додачу до "1 інструкція", щоб демонструвати заповнення/спорожнення конвеєра, forwarding і flush візуально — природне продовження поточної панелі регістрів/пам'яті
- Існуючі `cpu_tests` (перевіряють кінцевий стан після виконання) лишаються дійсними як тести на коректність результату; знадобляться нові тести окремо на цикловий рівень (кількість тактів, коректність forwarding, flush при branch)
