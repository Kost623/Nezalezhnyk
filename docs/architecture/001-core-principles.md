# ADR 001: Базові архітектурні принципи

Дата: 2026-07-11
Статус: Прийнято

## Контекст

Потрібно зафіксувати фундаментальні рішення на старті проекту, до яких прив'язані всі подальші розробки.

## Рішення

### 1. Власний відкритий ISA

Не використовуємо x86_64 чи ARM (обидва мають ліцензійні/патентні обмеження та історичний баласт закритих розширень). Розробляємо власний набір інструкцій з нуля.

Наслідок: Windows та macOS не зможуть запускатись без написання окремого порту ядра під наш ISA — це природний результат вибору архітектури, а не штучне обмеження.

### 2. Без прихованого рівня -3

Не реалізуємо аналог Intel ME / AMD PSP. Будь-яка функціональність з підвищеними привілеями (керування живленням, ініціалізація, low-level діагностика) реалізується як відкритий, аудитований код, що виконується в межах документованих і публічних рівнів привілеїв.

### 3. Енергетичний бюджет

Цільовий TDP: **до 40 Вт**. Це визначає:
- Обмежену кількість ядер на першій ітерації
- Помірну тактову частоту замість максимальної продуктивності
- Пріоритет енергоефективних мікроархітектурних рішень

### 4. Linux-first оптимізація

Первинна оптимізація системи команд і мікроархітектури орієнтована на патерни роботи ядра Linux (scheduler, memory management). Порти під інші ОС не є метою проекту, але й не блокуються штучно.

### 5. Архітектура пам'яті

Вирішено окремо в ADR 003: обрано модифіковану Harvard-архітектуру (розділення на рівні L1-кешу, спільна основна RAM).

## Наслідки

- Розробка ISA "з нуля" — довший шлях, ніж форк RISC-V, але дає повний контроль
- Потрібен власний тулчейн (асемблер, згодом компілятор) для нового ISA
- Реальне апаратне виробництво — довгострокова, не найближча мета; на першому етапі — програмний симулятор архітектури з графічною оболонкою
