# ADR 004: Розподіл opcode-простору

Дата: 2026-07-11
Статус: Прийнято

## Контекст

Поле `opcode` займає 7 біт (`[6:0]`) в усіх форматах інструкцій (ADR 002), що дає 2⁷ = 128 можливих значень. Потрібно зафіксувати структуру цього простору заздалегідь, щоб забезпечити місце для майбутніх розширень (плаваюча кома, векторні інструкції, спеціалізовані блоки для майбутніх GPU-подібних розширень) без ризику зламати сумісність базового набору.

## Рішення

Простір `opcode` ділиться на **4 групи по 32 значення**, за старшими двома бітами `opcode[6:5]`. Молодші 5 біт `opcode[4:0]` визначають конкретну інструкцію в межах групи.

| Група | opcode[6:5] | Діапазон | Призначення | Формати |
|---|---|---|---|---|
| 0 | `00` | 0–31 | Базова арифметика (ALU) | R, I |
| 1 | `01` | 32–63 | Робота з пам'яттю (load/store) | I, S |
| 2 | `10` | 64–95 | Керування потоком виконання | B, J |
| 3 | `11` | 96–127 | Зарезервовано для розширень | — (поки не визначено) |

Обґрунтування двоетапного декодування (спочатку `opcode[6:5]`, потім `opcode[4:0]`):
- Декодер може одразу за двома старшими бітами визначити, до якого функціонального блоку (ALU, контролер пам'яті, блок переходів) направити інструкцію — це перший, найшвидший рівень маршрутизації в апаратній логіці
- Явне резервування групи 3 гарантує, що майбутні розширення (FPU, вектори, кастомні прискорювачі — з перспективи майбутньої роботи над GPU) отримають власний ізольований під-простір, не конфліктуючи з базовим набором

## Наслідки

- Базовий набір інструкцій обмежений 96 opcode-значеннями (групи 0–2), що достатньо з запасом для мінімального робочого ISA
- Група 3 (96–127) залишається невизначеною до окремого майбутнього ADR, коли постане конкретна потреба в розширенні (наприклад, підтримка чисел з плаваючою комою)
- Розподіл є остаточним для базового ISA; зміна меж груп у майбутньому вважатиметься breaking change і потребуватиме окремого рішення про версіонування ISA
